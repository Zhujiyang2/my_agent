方案 B：分层上下文架构

三层分离：知识层（按需检索）、状态层（结构化快照）、流转层（近期对话+工具摘要）

整体架构
LLM Context Window（每次 API 调用）
📖 知识层 — 按需注入 1-3 条相关文档摘要（RAG打底）
📊 状态层 — 结构化 JSON：机器列表、任务状态、关键发现、上下文摘要
💬 流转层 — 最近 N 轮对话 + 工具调用摘要（非完整日志）
🗄️ 外存（不在窗口内）
📁 knowledge/ — 领域文档、经验规则
📁 .my_agent/state.json — 会话状态快照
📁 runs/ — 完整命令日志
🔧 新增工具
search_knowledge — 检索知识库
update_state — 更新状态快照
read_log — 按需读取完整日志

数据流：一次典型的多机训练调试交互

1. User: "node1训练OOM了，看看什么情况"
2. Agent: 调用 search_knowledge("NPU OOM") → 得到 2 条相关知识摘要 → 注入知识层
3. Agent: 调用 run_command("tail -100 log.txt", machine="node1")
4. run_command 返回: 日志写入 runs/xxx/node1.log，history 只保存摘要："OOM at layer 12, allocated 48GB/64GB"
5. Agent: 调用 update_state({"node1": "OOM at layer 12", "status": "debugging"})
6. Agent: 根据知识建议调整 batch_size，调用 run_command("修改后的训练命令", machine="node1")
7. 下一轮...状态层已经包含了 node1 的状态，知识层有相关经验
