# 昇腾NPU单机环境预检

当用户请求检查或查看 NPU/昇腾 环境状态时使用本技能。触发语义包括但不限于："检查NPU环境""查看NPU状态""NPU怎么样""昇腾环境ok吗""帮我看看npu"等。

## 检查维度

按以下维度逐项检查，缺一不可：

| 维度 | 内容 |
|------|------|
| 驱动与固件 | npu-smi 可用性、驱动版本、固件版本 |
| 芯片硬件 | 逐芯片：健康状态、温度、功耗、AI Core 使用率、HBM 使用量/总量 |
| 进程占用 | 逐芯片：运行中的进程名、PID、显存占用量，标注空闲卡 |
| 芯片间通信 | P2P 连通性矩阵、HCCS 状态 |
| 网络（可选） | RoCE/IB 网卡状态（单机场景非必需） |

## 可用工具

- `npu-smi info` — 芯片列表、基本信息
- `npu-smi info -t board -i <id>` — 单芯片板卡详情（驱动/固件版本）
- `npu-smi info -t usages -i <id>` — 温度、功耗、AI Core 使用率、健康状态
- `npu-smi info -t memory -i <id>` — HBM 内存使用
- `npu-smi info -t p2p -i <id>` — P2P 连通性矩阵
- `npu-smi info -t processes -i <id>` 或 `npu-smi info` 中的进程列表
- `ascend-dmi -i` — 诊断信息（补充验证）
- `ibstat` — RoCE/IB 网卡状态（可选）

## 约束

1. **动态发现芯片数量** — 从 `npu-smi info` 实际输出中提取芯片列表，遍历所有芯片。禁止假设只有 1 颗或固定数量
2. **不做伪判定** — 温度和 HBM 使用率仅报告原始数值，不做 PASS/FAIL 判断。仅当芯片健康状态字段明确非 OK 时才标记异常
3. **P2P 全量检查** — 检查所有芯片对之间的连通状态。若有不通的链路，指出具体是哪对芯片
4. **进程占用如实报告** — 列出每张卡上运行的进程（名称、PID、显存占用），明确区分空闲卡和占用卡
5. **RoCE 可选** — `ibstat` 不存在时标注 N/A，不算检查失败
6. **解析失败不中断** — 某个字段解析失败时，标注 WARN 并继续检查其余维度，不编造数值

## 输出

1. **终端表格** — 芯片硬件状态汇总表（健康/温度/功耗/AI Core/HBM）、进程占用表（NPU/进程/PID/显存），其余维度逐条列出结果
2. **JSON 报告** — 保存到 `./output/npu_check_<时间戳>.json`，包含所有维度的结构化数据
3. **关键解读** — 结尾用 2-4 条要点总结值得关注的发现（如有异常卡、空闲卡、P2P 不通等）。没有异常就如实说一切正常，不凑数
